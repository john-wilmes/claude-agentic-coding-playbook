## CRITICAL

### C1. md-size-guard uses `process.cwd()` instead of `hookInput.cwd`

**File**: `templates/hooks/md-size-guard.js:164`

```js
const cwd = process.cwd();
```

Every other hook that needs the project working directory reads `hookInput.cwd` from stdin (the hook input JSON). `md-size-guard` uses `process.cwd()` instead. When Claude Code invokes hooks, the process CWD may not match the project directory (hooks run from `~/.claude/hooks/`). This means `isMemoryFile()` and `checkClaudeSize()` will compute the wrong project key and silently fail to enforce limits. The MEMORY.md overflow protection — the hook's entire purpose — is unreliable.

### C2. md-size-guard emits malformed hook output

**File**: `templates/hooks/md-size-guard.js:178`

```js
console.log(JSON.stringify({ additionalContext: messages.join("\n\n") }));
```

All other hooks wrap their output in `{ hookSpecificOutput: { hookEventName: "PostToolUse", additionalContext: ... } }`. This hook emits `{ additionalContext: ... }` at the top level, missing the `hookSpecificOutput` wrapper and `hookEventName`. Claude Code will silently ignore the malformed output, so the user never sees MEMORY.md overflow warnings or CLAUDE.md size warnings. The hook writes the overflow file correctly but its feedback to the agent is dead.

### C3. install.sh uses `local` outside a function

**File**: `install.sh:1239`

```bash
local fleet_cron="5 6 * * * ..."
```

This line is inside an `if` block in the main script body, not inside a function. Under `set -euo pipefail` (line 3), `local` outside a function is an error in strict POSIX mode and causes undefined behavior in bash. In practice bash silently treats it as a global, but it violates the script's own discipline and will break if the block is ever refactored. More critically, some bash versions emit a warning to stderr which could confuse users during install.

### C4. context-guard install.sh comment claims "hard-blocks ALL tools" but code is pure pass-through

**File**: `install.sh:763-764` (comment in node -e block)

```
# PreToolUse entry: no matcher (hard-blocks ALL tools at 60%).
```

**File**: `templates/hooks/context-guard.js:170-174`

```js
if (isPreToolUse) {
  process.stdout.write(JSON.stringify({}));
  process.exit(0);
}
```

The install.sh comment says the PreToolUse mode "hard-blocks ALL tools at 60%", but the actual code is a pure pass-through that always returns `{}`. The code comments at lines 7-9 correctly say "Pure pass-through. Always returns {}. No hard blocks." The install.sh comment was never updated after the deadlock fix (documented in MEMORY.md lessons learned). Anyone reading settings.json or install.sh will believe there is a PreToolUse hard block when there is none. All enforcement is advisory PostToolUse only.

## HIGH

### H1. hooks/README.md is stale — lists only 9 hooks, 16+ exist

**File**: `templates/hooks/README.md`

The README table lists 9 entries. The directory contains 16 distinct hook/module scripts: `session-start.js`, `session-end.js`, `model-router.js`, `prompt-injection-guard.js`, `post-tool-verify.js`, `context-guard.js`, `stuck-detector.js`, `pre-compact.js`, `pre-commit`, `sanitize-guard.js`, `pii-detector.js`, `filesize-guard.js`, `bloat-guard.js`, `skill-guard.js`, `md-size-guard.js`, `pr-review-guard.js` (plus shared modules `log.js`, `bm25.js`, `knowledge-db.js`, `knowledge-capture.js`). Missing from the table: sanitize-guard, pii-detector, filesize-guard, bloat-guard, skill-guard, md-size-guard, pr-review-guard, and all shared modules. The README also incorrectly describes context-guard (line 14) as having "PreToolUse hard-blocks all tools when usage >= 60%", which contradicts the actual pass-through behavior.

### H2. README.md claims "~280 tests across 19 suites" — count is inaccurate

**File**: `README.md:120`

Running the actual hook test suites produces 300+ passing tests from hooks alone. The README also omits `tests/skills/*.test.sh` and `tests/skills/*.test.js` from the "run all at once" example command (lines 123-131), despite the project CLAUDE.md listing them as part of quality gates. The quality gates command in CLAUDE.md includes four `for` loops covering hooks, scripts, and skills tests, but the README shows only two.

### H3. README install tree shows config under `<install-root>/.claude/` — actually goes to `~/.claude/`

**File**: `README.md:53-74`

The "What Gets Installed" diagram shows everything under `<install-root>/.claude/`. But `install.sh:77` sets `GLOBAL_CLAUDE_DIR="$HOME/.claude"` and all hooks, skills, settings.json, and templates install there regardless of `--root`. The `--root` flag only controls the `research/` directory location (install.sh:420). The diagram is misleading about where configuration actually lives.

### H4. profiles/dev/ and profiles/research/ are orphaned — never installed, never tested

**Files**: `profiles/dev/`, `profiles/research/`

`install.sh:79` hard-codes `PROFILE_DIR="$SCRIPT_DIR/profiles/combined"`. The `dev` and `research` profiles are never referenced by install.sh, have no install path, and are not tested in CI. The README Architecture section (lines 189-199) describes all three profiles as if they are selectable, but no `--profile` flag exists. These profiles drift silently — `profiles/dev/skills/continue/SKILL.md` may be arbitrarily behind `profiles/combined/skills/continue/SKILL.md` with no mechanism to detect divergence.

### H5. No test for session-end.js git auto-commit logic

**File**: `tests/hooks/session-hooks.test.js`

The `session-hooks.test.js` file tests session-start behavior (18 tests) but session-end's core functionality — initializing `~/.claude` as a git repo, staging the correct memory path, committing, and pushing — is untested. This is the most side-effect-heavy hook in the system (it runs `git init`, `git add`, `git commit`, `git push`) and has zero test coverage on its primary behavior.

### H6. README says `--root` sends config to `<root>/.claude/` — it does not

**File**: `README.md:45`

```
| `--root <path>` | Install root directory (default: `~/Documents`). Config goes to `<root>/.claude/`, projects are siblings |
```

"Config goes to `<root>/.claude/`" is false. Config always goes to `~/.claude/`. The `--root` flag only determines where the `research/` directory is created (install.sh:420). The install.sh usage text (line 21) correctly says "The research/ dir goes here, projects are siblings" but the README table misrepresents it.

### H7. claude-loop.sh requires python3 but install.sh does not check for it

**File**: `scripts/claude-loop.sh:125,525,544`

The `log_event` function uses `python3` for JSON construction. Session timing uses `python3 -c "import time; ..."`. The `q` script also requires `python3` (line 12). But `install.sh:176-178` only checks for `git` and `node`. If python3 is missing, claude-loop fails at runtime with an unhelpful error. The `show_report` function (line 326) also uses python3.

### H8. CI runs only 2 of 19+ hook test suites

**File**: `.github/workflows/test-install.yml:624-636`

The `test-hooks` job runs only `session-hooks.test.js` and `model-router.test.js`. All other hook test suites (17+) are not executed in CI. The `test-install` job validates JavaScript syntax (`node --check`) for only session-start and session-end. A regression in prompt-injection-guard, context-guard, stuck-detector, or any other hook would not be caught by CI. The `test-investigate` job runs only the investigation scorer tests.

## MEDIUM

### M1. bloat-guard uses env vars for session ID — counter likely resets every call

**File**: `templates/hooks/bloat-guard.js:50-51`

```js
const sessionId = process.env.CLAUDE_SESSION_ID || process.env.PPID || "default";
```

Other hooks use `hookInput.session_id` from stdin JSON. `bloat-guard` reads from env vars that Claude Code may not set. `PPID` changes every invocation since hooks are spawned as child processes. This means the per-session new-file counter likely resets on every call, making the escalation threshold (5+ new files) unreachable in practice.

### M2. session-end.js auto-pushes to remote without user consent or opt-out

**File**: `templates/hooks/session-end.js:80-82`

```js
execFileSync("git", ["push"], { ...gitOpts, timeout: 8000 });
```

The SessionEnd hook auto-commits to `~/.claude` git repo and then pushes to remote on every session close. There is no opt-out mechanism, no environment variable check, and no documentation warning users about this behavior. If a user sets up a remote for `~/.claude`, their memory files are pushed automatically.

### M3. install.sh knowledge repo clone uses full history

**File**: `install.sh:1304`

```bash
git clone "$KNOWLEDGE_REPO" "$KNOWLEDGE_DIR"
```

The managed repos section (line 1230) correctly uses `--depth=1` for efficiency, but the knowledge repo clone does not. For large knowledge repos this downloads full history unnecessarily.

### M4. session-start.js exports dead-code functions `parseFrontmatter` and `scoreEntry`

**File**: `templates/hooks/session-start.js:44-92,304`

These functions are exported for testing and were the original entry scoring path. But the production code calls `knowledgeDb.queryRelevant()` which uses FTS5/SQL scoring. The old functions are dead code in production — only reached through test imports. They will silently diverge from the actual scoring logic.

### M5. pre-commit hook credential pattern has false positive risk on documentation

**File**: `templates/hooks/pre-commit:32`

```bash
grep -qEi '(password|secret|token|api_key|apikey)[[:space:]]*[=:][[:space:]]*["'"'"']?[^[:space:]"'"'"']{8,}' "$file"
```

This matches documentation like `password: "your-password-here"` or `api_key: placeholder_value` in README files, YAML examples, and config templates. The hook scans all staged text files with no extension-based exemption for `.md`, `.example`, `.sample`, or `.template` files.

### M6. README says `/checkpoint` runs quality gates — it does not

**File**: `README.md:83`

```
- **`/checkpoint`** -- Save all work, update memory with Current Work section, run quality gates, commit, push.
```

The actual `profiles/combined/skills/checkpoint/SKILL.md` has steps: update memory, commit and push, sync knowledge repo, verify. There is no "run quality gates" step. Quality gates were added to `/continue` (step 3.5D), not `/checkpoint`.

### M7. context-guard.js comment says "last 200KB" but code reads 512KB

**File**: `templates/hooks/context-guard.js:62,73`

```
 * Reads only the last 200KB to handle large transcripts efficiently.
...
const TAIL_SIZE = 512 * 1024;
```

The JSDoc says "last 200KB" but the constant is 512KB (512 * 1024). The code is correct for performance but the comment is stale.

### M8. knowledge-capture.js opens DB at module load time — stale HOME in tests

**File**: `templates/hooks/knowledge-capture.js:24-33`

The DB path is computed at `require()` time using `os.homedir()`. If `HOME` is changed after the module is loaded (e.g., in tests), the DB connection still points to the original HOME. The `session-start.js` works around this by computing `dbPath` at call time (line 110), but `knowledge-capture.js` does not.

### M9. promote/SKILL.md references obsolete filesystem knowledge entries path

**File**: `profiles/combined/skills/promote/SKILL.md:51,70-71`

```
Also check `~/.claude/knowledge/entries/` if that directory exists.
...
If `~/.claude/knowledge/entries/` exists (preferred path...):
```

The knowledge system migrated to SQLite. The promote skill still writes to the filesystem `entries/` directory as the "preferred path." These entries won't be found by session-start's SQLite-based injection unless the one-time migration re-runs.

### M10. create-project/SKILL.md uses different install root discovery than other skills

**File**: `profiles/combined/skills/create-project/SKILL.md:27-35`

Uses a hand-rolled upward directory walk to find `.claude/CLAUDE.md`. All other skills (`continue`, `investigate`, `playbook`) use the shared helper `bash ~/.claude/scripts/skills/find-install-root.sh`. This inconsistency means create-project could resolve a different install root.

### M11. investigate/SKILL.md references `scripts/check-citations.sh` with ambiguous relative path

**File**: `profiles/combined/skills/investigate/SKILL.md:306,337`

```
bash scripts/check-citations.sh "$INVESTIGATIONS_DIR" "{ID}"
```

The path `scripts/check-citations.sh` is relative. The actual file is at `profiles/combined/skills/investigate/scripts/check-citations.sh` in the repo, installed to `~/.claude/skills/investigate/scripts/`. If the agent runs this from the project CWD instead of the skill directory, the script won't be found.

### M12. Duplicate signal-state reset in claude-loop.sh

**File**: `scripts/claude-loop.sh:563-564,632-633`

`SIGNAL_RECEIVED` and `SIGNAL_COUNT` are reset at line 563 (after signal-death check) and again at line 632 (end of loop). The first reset clears signal state before the task-queue transition logic could consult it.

## LOW

### L1. Inconsistent stdin reading patterns across hooks

Some hooks call `process.stdin.resume()` before attaching listeners (context-guard, model-router, stuck-detector, session-start, session-end). Others omit it (filesize-guard, bloat-guard, skill-guard). The `on("data")` listener implicitly switches to flowing mode regardless, so both patterns work, but the inconsistency makes the codebase harder to audit.

### L2. bloat-guard.js and filesize-guard.js have unreachable `process.exit(0)` at module level

**Files**: `templates/hooks/bloat-guard.js:196`, `templates/hooks/filesize-guard.js:212`

Both hooks call `process.exit(0)` at the bottom of the `on("end")` handler AND at module level after the handler setup. The module-level `process.exit(0)` is unreachable because the `on("end")` handler always calls `process.exit(0)` first.

### L3. session-end.js shadows the `log` naming convention

**File**: `templates/hooks/session-end.js:18`

Defines `function log(msg)` which shadows the pattern used by all other hooks (`let log; try { log = require("./log"); }`). Renames the import to `logModule`. Works correctly but breaks naming consistency.

### L4. `q` script uses date-pinned model ID

**File**: `scripts/q:18`

```bash
DEFAULT_MODEL="claude-haiku-4-5-20251001"
```

The model ID includes a date stamp that will become stale. Other parts of the system use short names like `"haiku"`, `"sonnet"`, `"opus"`. The raw API requires full model IDs, but the pinned version should be periodically reviewed.

### L5. Inconsistent use of `console.log` vs `process.stdout.write` for JSON output

Most hooks use `process.stdout.write(JSON.stringify(...))`. `md-size-guard.js` uses `console.log(JSON.stringify(...))` which appends a newline. Both work for JSON parsing but mixing patterns makes auditing harder.

### L6. templates/registry/mcp-servers.json content not validated by tests

The install.sh reads this file and injects entries into settings.json. No test validates the JSON structure or that the MCP server configurations are well-formed. A malformed entry would silently propagate.

### L7. profiles/research/README.md and eval ground-truth files suggest active maintenance

**Files**: `profiles/research/README.md`, `profiles/research/eval/ground-truth/*.json`

These files imply the research profile is maintained, yet it is never installed. This creates confusion about whether the profile is supported or abandoned.

### L8. CONTRIBUTING.md and CODE_OF_CONDUCT.md not audited for accuracy

These exist but their content was not verified against current project practices (testing instructions, contribution workflow, etc.).

### L9. README Benchmarks section references `scripts/swe-bench.sh` costs with dollar estimates

**File**: `README.md:209-213`

Claims "~$10-25 in API costs" for 5 tasks and "~$100-250" for 25 tasks. These estimates are not verified and depend on model pricing that may have changed since they were written.

### L10. fleet-index.test.js exists but fleet-index.js is in templates/fleet/, not templates/hooks/

**Files**: `tests/hooks/fleet-index.test.js`, `templates/fleet/fleet-index.js`

The test file is in `tests/hooks/` following the pattern for hooks, but the source file is in `templates/fleet/`. This is a naming/organization inconsistency — the test works (it imports from the correct path) but the location is misleading.

**Highest-impact findings**: The md-size-guard bugs (C1, C2) render MEMORY.md overflow protection non-functional — the hook truncates the file but never tells the agent it did so, and may compute the wrong project path. The stale documentation (H1, H3, H6, M6) gives users a false picture of what is installed and how features work. The CI coverage gap (H8) means the vast majority of hook logic is not tested in CI. The orphaned dev/research profiles (H4) create a maintenance burden with no install path and diverge silently from the combined profile that is actually used.
