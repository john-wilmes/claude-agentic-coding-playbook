## HIGH SEVERITY

### 1. `bloat-guard.js` â€” Throwaway-file check does NOT count toward session state

**File:** `/home/johnw/Documents/claude-agentic-coding-playbook/templates/hooks/bloat-guard.js`
**Lines:** 122â€“147

When `isThrowaway(filename)` returns true, the function returns early with a warning **without** adding the file to `createdFiles` or calling `saveState()`. This means throwaway files are never tracked in the session counter, so `ESCALATION_THRESHOLD` is never incremented for them. A session that creates 6 throwaway files in a row will warn on each individually, but the escalation message ("X new files created this session") will never fire.

Additionally, the early return at line 123 means the throwaway file path is not persisted even if the Write proceeds (the warn decision does not block). The counter then diverges from reality.

**Fix:** Call `saveState` and increment the counter before the throwaway check's early return, or restructure so escalation accounts for all new files including throwaway ones:

```js
// After the /tmp/ exemption check, always track:
const createdFiles = loadState();
createdFiles.push(expanded);
saveState(createdFiles);
const count = createdFiles.length;

if (isThrowaway(filename)) { ... }   // still warn, but count is already tracked
if (count > ESCALATION_THRESHOLD) { ... }
```

## MEDIUM SEVERITY

### 2. `bloat-guard.js` â€” `getStateFile()` uses `PPID` as session ID fallback

**File:** `/home/johnw/Documents/claude-agentic-coding-playbook/templates/hooks/bloat-guard.js`
**Line:** 50

```js
const sessionId = process.env.CLAUDE_SESSION_ID || process.env.PPID || "default";
```

`PPID` is the parent process ID of the Node process, not the Claude session ID. All hooks for the same session will have the same `PPID` value only by accident â€” in practice, each `node` invocation spawned from the hook runner will have the hook runner as its parent, which may or may not be consistent across tool calls. This can cause session state to bleed between unrelated sessions or to be lost entirely.

All other hooks use `hookInput.session_id` (parsed from stdin), not an env var. `bloat-guard.js` should follow the same pattern.

**Fix:**
```js
// In the stdin handler, after parsing hookInput:
const sessionId = event.session_id || "unknown";
// Pass sessionId to getStateFile(), loadState(), saveState()
```

### 3. `sanitize-guard.js` â€” `Bash` command scanning for PII is a false-positive magnet

**File:** `/home/johnw/Documents/claude-agentic-coding-playbook/templates/hooks/sanitize-guard.js`
**Line:** 147

```js
if (toolName !== "Edit" && toolName !== "Write" && toolName !== "Bash") {
```

Scanning `Bash` command strings for PII (e.g., SSNs, phone numbers) is likely to produce false positives on innocuous shell commands that happen to match patterns (e.g., version strings matching `PHONE_US`, numeric arguments matching SSN format). Worse, when PII is detected in a Bash command, the hook emits a `deny` with "Use this redacted version instead" â€” but the redacted version will be a shell command with placeholders substituted, which will be syntactically broken.

For example: `openssl rand -hex 123-45-6789` â†’ denied with redacted: `openssl rand -hex [SSN]`.

**Fix:** Remove `Bash` from the PreToolUse path. Bash commands should only be scanned in PostToolUse (after they produce output), not before execution. The PostToolUse path already handles command output.

### 4. `prompt-injection-guard.js` â€” `cat .env` pattern is too narrow

**File:** `/home/johnw/Documents/claude-agentic-coding-playbook/templates/hooks/prompt-injection-guard.js`
**Line:** 14

```js
{ pattern: /cat\s+\.env\b/, reason: "Potential credential exfiltration: reading .env file" },
```

This only catches `cat .env` with a word boundary. It misses:
- `cat .env.local`, `cat .env.production` (common variants)
- `cat $PWD/.env` (absolute path)
- `type .env` (Windows-style but occasionally used in WSL)
- `less .env`, `more .env`, `head .env` (the filesize-guard already blocks some, but it allows small files)

Since the codebase already checks for `.env` reads in filesize-guard, this is defense-in-depth â€” but the narrow match gives false confidence.

**Fix:** Broaden to `/cat\s+\.env(?:\.\w+)?$|\bcat\s+[^ ]*\/\.env\b/i` or, better, add `.env*` to the filesize-guard's explicit blocklist instead.

### 5. `context-guard.js` â€” `output_tokens` excluded from context total

**File:** `/home/johnw/Documents/claude-agentic-coding-playbook/templates/hooks/context-guard.js`
**Lines:** 119â€“122

```js
const totalTokens = (usage.input_tokens || 0) +
                    (usage.cache_read_input_tokens || 0) +
                    (usage.cache_creation_input_tokens || 0);
```

`output_tokens` are not included. However, Claude Code's 200K context window counts the running total of both input and output tokens as the conversation grows. Omitting output tokens underestimates usage. For a session with many long agent responses, this can be significant (5â€“15% underestimate).

The comment at line 117 says "total context = all three summed" â€” this is accurate for the input side but output tokens from prior turns re-enter as input on subsequent turns. The current formula is consistent with what the transcript reports as `input_tokens` for the *current* turn (which already includes cached output from prior turns via `cache_read_input_tokens`). However, the uncached `output_tokens` of the current turn are not yet in input.

**Note:** This may be intentionally conservative (the hook is advisory). Still worth documenting more clearly or adding `output_tokens` with a note.

### 6. `post-tool-verify.js` â€” `STATE_FILE` shared across concurrent sessions creates write contention

**File:** `/home/johnw/Documents/claude-agentic-coding-playbook/templates/hooks/post-tool-verify.js`
**Line:** 22

```js
const STATE_FILE = path.join(os.homedir(), ".claude", ".verify-last-run");
```

The comment on lines 20â€“22 acknowledges this is intentional for cross-session sharing of `lastPassed`/`lastFailOutput`. However, the update path (lines 119â€“127) does a read-then-write non-atomically: if two hooks fire simultaneously for different sessions, one write can overwrite the other. On a machine with multiple open Claude projects, this could corrupt the `lastPassed` state used for knowledge-candidate detection.

**Fix:** Use `fs.writeFileSync` with a tmp file + atomic rename, or accept the race as low-impact (knowledge capture just fires spuriously).

### 7. `session-start.js` â€” `knowledgeDb.openDb()` called without arguments on line 174

**File:** `/home/johnw/Documents/claude-agentic-coding-playbook/templates/hooks/session-start.js`
**Line:** 174

```js

```

All other callers (line 111, `knowledge-capture.js`) pass an explicit `dbPath`. Calling `openDb()` with no arguments relies on the function's default behavior. If `knowledge-db.js` doesn't define a sensible default, this silently fails. The `try/catch` wrapper absorbs the error, but knowledge import from JSONL is silently skipped without any log entry.

### 8. `claude-loop.sh` â€” `CLAUDE_LOOP_PID` exported as the `claude-loop` script's PID, but written as `$$` from subshell

**File:** `/home/johnw/Documents/claude-agentic-coding-playbook/scripts/claude-loop.sh`
**Lines:** 533â€“537

```bash
(
  echo $BASHPID > "${CLAUDE_PID_FILE}"
  export CLAUDE_LOOP_PID=$$
  export CLAUDE_LOOP_SENTINEL="${SENTINEL_FILE}"
  ...
)
```

Inside the subshell, `$$` refers to the **parent shell's PID** (bash subshells inherit `$$` from their parent). So `CLAUDE_LOOP_PID` is set correctly to the outer claude-loop PID. However, `CLAUDE_PID_FILE` is written with `$BASHPID`, which is the subshell's PID. These are different values. The sentinel watcher reads `CLAUDE_PID_FILE` to get the `claude` process PID. If `claude` is launched via `exec` inside the subshell, then `$BASHPID` will be replaced by `claude`'s PID at `exec` time â€” this is correct.

But `CLAUDE_LOOP_PID` (used by hooks to derive the sentinel path) gets the outer PID (correct), while `CLAUDE_PID_FILE` gets the pre-exec subshell PID (also correct since exec replaces it). The design is correct but the different PID sources in the same block make it confusing and fragile. A comment explaining this invariant would prevent future breakage.

**Recommendation:** Add inline comment:
```bash
# CLAUDE_LOOP_PID = outer script PID (stable for loop lifetime, used for sentinel path)
# BASHPID before exec = subshell PID, replaced by claude's PID at exec time
```

### 9. `install.sh` â€” Settings.json write is not atomic

**File:** `/home/johnw/Documents/claude-agentic-coding-playbook/install.sh`
**Lines:** ~489â€“509, and repeated for each hook**

Each hook's settings configuration does a read-parse-modify-write of `settings.json` in a separate `node -e` invocation. If `install.sh` is interrupted between writes, `settings.json` can be left in a partially-configured state with no indication to the user. The file is also not backed up before modification.

**Fix:** Consolidate all `settings.json` modifications into a single Node script invoked at the end of install, or use a tmp file + atomic rename for each write.

### 10. `md-size-guard.js` â€” Uses `console.log` instead of `process.stdout.write` inconsistently

**File:** `/home/johnw/Documents/claude-agentic-coding-playbook/templates/hooks/md-size-guard.js`

All other hooks wrap in `hookSpecificOutput: { hookEventName: "...", additionalContext: "..." }`. Check whether the unwrapped form is intentional or a schema mismatch.

### 17. `bloat-guard.js` â€” `/tmp/` exemption uses `os.tmpdir()` which may differ from `/tmp/`

**File:** `/home/johnw/Documents/claude-agentic-coding-playbook/templates/hooks/bloat-guard.js`
**Lines:** 115â€“117

```js
if (expanded.startsWith("/tmp/") || expanded.startsWith(os.tmpdir())) {
  return null;
}
```

On most Linux systems `os.tmpdir()` returns `/tmp`, making this redundant. On macOS, `os.tmpdir()` returns `/var/folders/...` (a temp directory), not `/tmp`. A file at `/tmp/foo.js` on macOS would NOT be exempted by the `os.tmpdir()` check, but WOULD be by the `/tmp/` string check. This is likely correct but the logic order could be consolidated as `expanded.startsWith(os.tmpdir()) || expanded.startsWith("/tmp/")`.

### 18. `pre-compact.js` â€” Regex for replacing existing snapshot section is greedy

**File:** `/home/johnw/Documents/claude-agentic-coding-playbook/templates/hooks/pre-compact.js`
**Line:** 80

```js
/## Pre-compact snapshot[\s\S]*?(?=\n(?:##? )|\s*$)/
```

The `[\s\S]*?` is non-greedy and the lookahead `(?=\n(?:##? )|\s*$)` stops at the next section header. However, the `\s*$` branch matches "only whitespace to end of string", which includes all remaining content if the snapshot is the last section. This is the intended behavior, but if MEMORY.md ends without a trailing newline and the snapshot is not the last section, the regex may not match, silently leaving the stale snapshot in place.

## SECURITY

### 23. `prompt-injection-guard.js` â€” `rm -rf` guard can be bypassed with unusual flag ordering

**File:** `/home/johnw/Documents/claude-agentic-coding-playbook/templates/hooks/prompt-injection-guard.js`
**Lines:** 52â€“56

The `rm -rf` guard checks for flags with:
```js
/\brm\s+(?:\S+\s+)*-[a-zA-Z]*r[a-zA-Z]*f/.test(cmd)
```

This matches common forms but misses:
- `rm --recursive --force /` (long flags)
- `rm -r --force /` (mixed short/long)
- `\rm -rf` (backslash disables alias, still executes)

The blocking condition also requires specific dangerous targets (`/`, `~/`, `/*`, `~/\*`, `sudo rm`). The pattern `rm -rf /home/user/important/` would NOT be blocked. This is by design (the guard focuses on catastrophic broad-scope deletions), but users should understand the guard's limited scope.

### 24. `q` script â€” Request body passes `PROMPT` via Python argv (safe), but `SYSTEM` prompt is unvalidated

**File:** `/home/johnw/Documents/claude-agentic-coding-playbook/scripts/q`
**Lines:** 125â€“140

The `q` script properly passes `SYSTEM` and `PROMPT` as Python `sys.argv` arguments (avoiding shell injection via Python). This is correct. However, `SYSTEM` is user-supplied via `--system` without any length check or sanitization. A very long system prompt (e.g. 1MB pasted text) could cause the curl request to exceed API limits silently. Low severity since this is a CLI tool for trusted users.
